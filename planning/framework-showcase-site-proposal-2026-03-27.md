# Pulp Framework Showcase Site Proposal

Date: 2026-03-27

## Purpose

Pulp needs a public-facing surface that does for the framework what the Visage showcase does for Visage:

- make the platform legible in minutes
- show that the UI system is real
- show that animation and theming are first-class
- show that the AI-designer workflow produces shippable results

This should not replace the local-first docs system. It should be the public presentation layer built on top of it.

Claude alignment:

- Claude’s highest-signal recommendations for the public face were: a primitives gallery, an animation playground, a theme switcher, an AI restyling demo, and an explicit ship-to-plugin pipeline. This proposal uses that structure.

## Guiding Principle

The site should not read like a static marketing page with a few screenshots. It should function like an evidence-backed product surface:

- every section demonstrates a real capability
- examples are backed by repo-owned source
- the same examples should be runnable locally
- docs, screenshots, videos, and status labels should all derive from the same underlying examples and manifests

## What The Site Should Prove Immediately

Within a few minutes, a visitor should understand:

- Pulp ships a real native UI framework, not a browser shell
- the framework has serious audio-native primitives
- themes and style packs are structured data
- JS authoring and hot reload are real
- animations are built in, theme-aware, and testable
- AI restyling produces durable plugin-owned results
- example plugins can actually be built, previewed, and shipped

## Proposed Site Structure

### 1. Landing Page

Goal: one-screen explanation of what Pulp is and why it is differentiated.

Must show:

- native UI + audio framework positioning
- local-first AI design workflow
- real plugin examples
- a concise feature grid
- immediate links to interactive galleries and example workflows

### 2. Primitives Gallery

Goal: show all shipped UI primitives in one place.

Should include:

- knobs, faders, toggles, buttons
- ComboBox, tabs, text editor, panels, scroll surfaces
- meters, waveform, spectrum, XY pad
- future audio-native controls like envelopes, keyboards, modulation overlays, routing surfaces

Each primitive card should show:

- live or recorded interaction
- available styles
- important props/tokens
- JS example
- screenshot reference

### 3. Motion Gallery

Goal: make animation feel like a platform feature, not an implementation detail.

Should include:

- hover and press behavior
- panel reveal/dismiss transitions
- scroll behavior
- meters and live signal motion
- token-driven motion presets

This page should also explain:

- FrameClock
- motion tokens
- deterministic testing

### 4. Theme And Style Pack Explorer

Goal: show that the same UI can be materially restyled without rewriting it.

Should include:

- side-by-side themes on the same example UI
- token groups: color, spacing, radius, typography, motion
- before/after diffs
- downloadable or inspectable style-pack JSON

This page should make it obvious that style is data, not scattered widget code.

### 5. AI Designer Workflow Demo

Goal: prove the differentiated story.

This should walk through:

1. starting from a real example plugin UI
2. prompting with natural language
3. applying token/style changes live
4. inspecting the produced diff
5. accepting and saving the style pack
6. reopening the plugin and seeing the same locked-in result

If this page is excellent, it will do more for Pulp than a large amount of generic framework prose.

### 6. Plugin Showcase

Goal: show that Pulp’s UI system is not isolated from real products.

Each example plugin page should show:

- what the plugin does
- what Pulp features it demonstrates
- screenshots and short recordings
- parameter/UI binding story
- supported formats/platforms
- style packs or themes available for it

This should include at least one page that explicitly shows:

- restyle plugin
- save style
- rebuild
- ship

### 7. Authoring Surface

Goal: help developers understand how they actually build with the system.

Should show:

- JS authoring surface
- bridge model
- token-diff model
- inspector/debug tooling
- local-first workflow

This is where short source snippets matter more than long API dumps.

### 8. Validation And Shipping Page

Goal: show maturity and trust.

Should show:

- screenshot regression pipeline
- animation tests
- token round-trip tests
- plugin validation
- what “supported” vs “experimental” means

This page can prevent the showcase from feeling like a glossy layer on top of weak delivery.

## Presentation Style

The site should feel designed, not autogenerated.

Recommended qualities:

- bold visuals, not default-docs chrome
- strong typography
- motion used sparingly but intentionally
- dark/light/theme switching where helpful
- embedded short videos or image sequences for interaction-heavy examples
- a consistent information architecture so visitors can browse or dive deep

It should feel closer to a product showcase plus manual than a traditional API site.

## Source Of Truth Model

The public site should not become the source of truth.

Preferred model:

- repo-owned examples drive screenshots and recordings
- repo-owned docs drive narrative text
- repo-owned manifests drive status/support labels
- the site is a build target from those inputs

This keeps local and hosted docs aligned.

## Content Pipeline Recommendation

For each showcased capability:

1. have a canonical repo example
2. have a screenshot or capture script
3. have a short explanation page
4. have source snippets or generated reference
5. publish the same artifact locally and online

That prevents “site drift,” which is exactly the credibility problem Pulp is trying to eliminate elsewhere.

## First Showcase Releases

The first public version does not need everything. It should focus on the pieces that best express the platform.

Recommended initial release:

- landing page
- primitives gallery
- motion gallery
- theme/style-pack explorer
- AI-designer workflow demo
- 2 to 3 example plugin pages

## Success Criteria

The site is succeeding if a skeptical engineer can answer these questions quickly:

- What can Pulp build today?
- What does the UI system actually look like?
- How good are the animations and controls?
- Is the design system real or just cosmetic?
- Can I build and ship an actual plugin with this?
- What is meaningfully different here from JUCE, iPlug, or Visage?

## Bottom Line

Pulp should have a public showcase site.

Not because “frameworks need websites,” but because this framework’s strongest qualities are experiential:

- visual quality
- motion
- audio-native controls
- AI-guided restyling
- plugin-shipping workflow

Those qualities are hard to communicate through reference docs alone. A strong showcase site would make the framework dramatically easier to understand, trust, and want to try.
