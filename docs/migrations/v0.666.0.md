## Behavior change — `signal::Oversampler` Biquad lane

The default `fir_biquad` anti-aliasing lane gave every 2x stage the same
absolute cutoff (`0.4 * base_sample_rate`) regardless of which stage was
filtering. That had two consequences this release corrects, so the lane's
rendered output changes at some factors.

- **`Factor::x2` is bit-identical.** The default configuration
  (`fir_biquad` + `x2`) is unchanged, byte for byte. If that is what your
  plugin uses, nothing about your audio moves.
- **`Factor::x4` gain is fixed.** A legacy direct-4x path folded unsuppressed
  images coherently onto DC, adding a broadband **+0.11 dB** gain error
  (DC gain 1.0128). x4 output now sits at unity to within a float ULP — so
  x4 output *changes* (that is the fix).
- **`Factor::x8` / `x16` passband is flat again.** Cascading same-cutoff
  biquads dragged the passband down to roughly **-8.9 dB at 0.3 of the base
  rate** at x16. Each stage now filters at its own rate, so the passband is
  flat to within ~0.2 dB across every factor.

### The honest tradeoff at x8/x16

Flattening the passband gives back the high end the old code was quietly
removing, so **measured alias rejection at x8/x16 drops** (worst case about
26 dB down to about 8 dB at x16). That is not a regression in quality: the
old rejection number was bought entirely by lowpassing away the top of the
audio band — you would get the same figure from a -9 dB shelf. The Biquad
tier's honest worst-case rejection is ~7 dB at *every* factor; raising the
factor buys the callback's own harmonics room to fold, not a better filter.

**If real anti-aliasing matters, do not reach for a higher factor on the
Biquad lane** — use `Kind::polyphase_iir` (flat passband, minimum-phase) or
`Kind::linear_phase_fir` (96 dB / 140 dB, host-compensated latency). This was
already true; this release just stops the Biquad lane from hiding it behind a
muffled top end.

### Unrelated performance fix in the same release

A `std::function` callback passed to `Oversampler::process()` /
`process_block()` is now taken by reference instead of by value, so it is no
longer copied once per sample (a 256-sample block previously heap-allocated
hundreds of times off the audio thread). This does not change rendered
output — except that a `std::function` wrapping a callable with mutable
by-value state now keeps that state across samples instead of silently
discarding it on every copy.

## Migration steps

- No API change and no code edits required.
- If you render golden/reference audio through `fir_biquad` at `x4`, `x8`,
  or `x16`, regenerate your references against this release.
- If you were using a high `fir_biquad` factor for its aliasing behavior,
  switch to `Kind::polyphase_iir` or `Kind::linear_phase_fir` instead.

Full rationale lives in `docs/guides/signal-processing.md` (the "Oversampler"
section).
